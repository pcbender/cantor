# Canto Memory Service Implementation Plan

Status: Approved
Workstream: CP-1600
Design source: `docs/Canto Memory Service Specification.md`

## Objective

Implement a local, governed Memory Service that lets Canto Workers retrieve
approved context and propose durable knowledge without creating a second
Approval model, execution path, registry, or evidence store.

CP-1601 is the approved scope and architecture packet. The approved
implementation sequence begins with CP-1602 and ends with CP-1614.

## Frozen Decisions

- `MemoryService` owns memory lifecycle, retrieval, conflict handling, and
  source-reference rules.
- `StateStore` owns durable records and atomic transitions.
- Existing Approval objects remain the only persisted approval authority.
- Initial durable scopes are `global:terminology`, `project:<project_id>`, and
  `repo:<repo_id>`.
- Project identity is Canto-generated and links explicit repository identities.
- Read-only inspection must not migrate state, initialize WAL files, or require
  writable state.
- Retrieval is deterministic, local, lexical, and budgeted. It performs no
  model or network calls.
- Secrets are rejected before memory text or unsafe event payloads are stored.
- Existing Job, Plan, Approval, Delegation, artifact, repository, commit, and
  Graphify records remain authoritative evidence. Memory stores typed pointers.
- The frozen orchestration HTTP contract is unchanged.
- Daemon, MCP, vector search, cloud sync, multi-user authority, and broad global
  preferences remain deferred.

## Implementation Order

### CP-1601 - Memory Service Scope and Architecture

Status: complete when the approved design specification is accepted.

Deliverables:

- approved memory classes, types, statuses, scopes, and authority model;
- approved budget, retention, conflict, and project-identity defaults;
- explicit MVP and deferred boundaries.

Acceptance:

- design is approved;
- no runtime behavior changes;
- implementation packets preserve the existing Approval and execution models.

### CP-1602 - StateStore Schema Migration Framework

Objective: Allow additive SQLite schema upgrades without resetting existing
single-user state.

Deliverables:

- ordered, transactional schema migrations from the current schema version;
- migration metadata and recoverable failure behavior;
- compatibility behavior for read-only StateStore access;
- migration contract tests using an existing version-1 database fixture.

Acceptance:

- an existing state database migrates in place;
- a failed migration leaves the prior database recoverable;
- unknown future schema versions fail clearly;
- read-only access never runs migrations or creates WAL files;
- existing StateStore tests pass.

### CP-1603 - Memory Domain Models and Persistence

Objective: Add memory, project, link, alias, tag, and event records behind the
StateStore boundary.

Deliverables:

- Pydantic models for MemoryItem, MemoryEvent, MemoryLink, MemoryAlias,
  MemoryTag, MemoryProject, and project repository membership;
- status transition definitions and confidence values;
- SQLite tables, indexes, and narrowly scoped StateStore methods;
- MemoryStateStore parity for unit tests;
- Redis compatibility decision documented and implemented only where required
  by the retained StateStore contract.

Acceptance:

- records round-trip through supported stores;
- illegal transitions fail without mutation;
- scope, provenance, timestamps, confidence, and source fields are required;
- repository and project identifiers are not inferred from names or paths.

### CP-1604 - MemoryService Lifecycle and Secret Rejection

Objective: Centralize memory lifecycle rules before exposing CLI writes.

Deliverables:

- MemoryService create, propose, supersede, expire, delete, purge, and event
  operations;
- atomic activation and supersession primitives;
- pre-persistence secret scanner for body, source metadata, and event payloads;
- safe errors and redacted rejection events.

Acceptance:

- callers cannot bypass lifecycle transitions through public service methods;
- secret-like values are rejected before unsafe text is persisted;
- purge requires an explicit reason;
- normal deletion is soft and auditable;
- service tests cover concurrent transition attempts.

### CP-1605 - Project Identity and Repository Membership

Objective: Support project memory across explicitly linked bootstrapped
repositories.

Deliverables:

- project create, list, show, link-repository, and unlink-repository service
  operations;
- matching `canto memory project ...` CLI commands;
- audited membership changes using existing `repo_id` identity;
- scope eligibility resolver for global terminology, project, and repository
  scopes.

Acceptance:

- project IDs are generated by Canto and durable;
- duplicate labels do not collide;
- linking requires a valid bootstrapped repository identity;
- membership eligibility does not override task-granted memory scopes;
- removing membership prevents future project-memory retrieval for that repo.

### CP-1606 - Approval Subject Linkage and Activation

Objective: Reuse the existing Approval object for memory activation and
rejection.

Deliverables:

- additive, compatibility-reviewed Approval subject identity for memory;
- memory approval request creation;
- dispatch from existing `canto approve` and `canto reject` paths;
- atomic Approval and MemoryItem state transitions;
- retention rejection for expired pending proposals.

Acceptance:

- no MemoryApproval model or `canto memory approve/reject` commands exist;
- approving a memory subject activates exactly that proposal;
- rejection preserves proposal and audit history;
- concurrent decisions produce one terminal result;
- Job and Plan approval behavior remains unchanged.

### CP-1607 - Memory Administration CLI

Objective: Expose the governed lifecycle through a consistent local CLI.

Deliverables:

- `canto memory status`;
- propose, list, show, request-approval, supersede, expire, delete, and purge;
- structured JSON output where existing CLI conventions support it;
- concise validation and state-access errors without tracebacks.

Acceptance:

- rejected, expired, superseded, and deleted items are hidden by default;
- explicit status flags expose retained administrative records;
- Worker invocation context cannot claim Developer authority through CLI flags;
- read-only commands work against read-only state.

### CP-1608 - Scoped Recall and Budget Profiles

Objective: Implement deterministic lexical retrieval with explicit scope and
budget enforcement.

Deliverables:

- SQLite FTS5 index maintenance and fallback behavior;
- `canto memory recall`;
- type, status, freshness, and scope filters;
- deterministic estimated-token accounting;
- named profiles: startup, resolve-reference, review, and planning.

Acceptance:

- recall performs no network or model calls;
- disallowed scopes never appear;
- default recall returns active memory only;
- item and token ceilings are enforced deterministically;
- callers may lower but not silently raise policy ceilings.

### CP-1609 - Reference Resolver, Aliases, and Conflicts

Objective: Resolve project terminology without guessing and identify explicit
conflicts conservatively.

Deliverables:

- `canto memory resolve`;
- normalized alias and terminology lookup;
- ambiguity and unresolved-reference results;
- deterministic `conflicts_with` links from aliases, explicit links, and exact
  normalized reference matches;
- atomic approve-as-supersession flow.

Acceptance:

- resolver returns scope, source, freshness, confidence, and status;
- ambiguous references are reported rather than guessed;
- conflict detection does not invoke a model;
- conflict discovery never silently selects an authoritative item.

### CP-1610 - Context Pack Assembly

Objective: Produce curated memory bundles for Developer planning and Worker
execution.

Deliverables:

- `canto memory context-pack`;
- deterministic sections for glossary, decisions, constraints, outcomes, known
  traps, open questions, and source pointers;
- task/delegation-aware scope resolution;
- all four approved default budget profiles.

Acceptance:

- context packs respect both membership eligibility and assigned scopes;
- raw transcripts, large logs, source files, and full Graphify data are absent;
- repeated requests against unchanged state are stable enough for fixtures;
- read-only assembly does not mutate state.

### CP-1611 - Governed Record Source References

Objective: Link observations and outcomes to existing Canto evidence without
copying that evidence into memory.

Deliverables:

- typed source-reference validation for Job, Plan, Approval, DelegationTask,
  ExecutorSession, ExecutorLaunch, DelegationResult, artifact, repository path,
  and commit records;
- attach-observation and attach-outcome service/CLI operations;
- existing Worker/model provenance included when available.

Acceptance:

- missing local Canto record IDs fail clearly;
- historical repository/external pointers may remain explicitly unresolved;
- observations remain non-authoritative;
- governed outcomes retain source freshness and commit/artifact identity;
- no source payload is duplicated wholesale.

### CP-1612 - Retention, Export, and Audit

Objective: Give the Developer deterministic ownership and cleanup controls.

Deliverables:

- retention processor for 7-day low-value observations, 30-day observations,
  90-day summaries and proposals, and indefinite governed outcomes/audit;
- `canto memory export` and `canto memory audit`;
- deterministic selectors by scope, project, repo, type, status, and date;
- explicit inclusion flags for rejected, superseded, expired, and deleted items.

Acceptance:

- pending proposal expiry atomically rejects its pending Approval with a system
  retention reason;
- retention does not infer low value with a model;
- exports are deterministic and exclude deleted records by default;
- normal lists hide rejected proposals while audit preserves them.

### CP-1613 - Graphify Pointers and Agent Guidance

Objective: Integrate memory usage into Canto-enabled repositories without
turning Graphify or AGENTS.md into the memory store.

Deliverables:

- opaque Graphify snapshot/node/community/report source pointers;
- AGENTS and generated Canto role guidance for resolve/context-pack usage;
- repo doctor checks for generated guidance compatibility;
- documentation for stale or unavailable Graphify references.

Acceptance:

- missing local graph output does not invalidate historical pointers;
- no full Graphify graph is copied into memory;
- existing AGENTS.md content is not clobbered;
- bootstrap remains free of secrets and durable memory data.

### CP-1614 - End-to-End Dogfood, Security, and Stability

Objective: Prove the complete local single-user Memory Service flow before any
MCP or server work.

Deliverables:

- integration test covering project creation, proposal, Approval, recall,
  resolution, context pack, source attachment, supersession, retention, export,
  and audit;
- multi-process SQLite and read-only-state tests;
- secret/redaction regression tests;
- README, status, roadmap, CLI help, and local installation updates;
- a Canto-on-Canto dogfood demonstration using repository memory only.

Acceptance:

- the full flow uses the existing StateStore and Approval paths;
- read-only inspection works without writable `~/.canto` state;
- no orchestration HTTP contract fields or endpoints change;
- no daemon, MCP, vector database, cloud sync, or model-based retrieval appears;
- full test suite passes.

## Stop Conditions

Pause implementation when any packet requires:

- a second approval object or execution path;
- a breaking orchestration contract change;
- an unresolved StateStore migration or rollback risk;
- a new multi-user, organization, daemon, MCP, vector, or cloud-sync design;
- silent scope widening or model-based authority decisions.

Stop after CP-1614 for Developer review.
